# Lab: observability & performance

This lab is inverted. Every other lab ships stubs you fill in; this one ships
a complete, working service that is deliberately slow. You profile it, name
the bottleneck, fix it, and prove the fix with numbers measured against the
shipped baseline. Nothing here is broken. It is just slow, which in Tier 3 is
the same thing.

## The shape

- `svc/baseline.go` is the shipped hot path, `Baseline`. It renders parsed
  access-log entries into the plain-text report served at `/process`, and it
  builds that report with string concatenation. Strings in Go are immutable:
  every `+=` allocates a new string and copies the whole report accumulated so
  far, so rendering n entries copies O(n^2) bytes. Do not edit this file; it
  is the anchor everything is measured against, and a golden test pins it.
- `svc/optimized.go` is yours. `Optimized` starts as an exact copy of
  `Baseline`: same output, same quadratic cost. This is the only file you
  change.
- `main.go` runs the service: the report on `127.0.0.1:8080/process`, pprof
  on the admin port `127.0.0.1:6060`.
- `loadgen/` is a small stdlib load generator, so the lab needs no external
  tool. If you have `hey` or `ab`, they do the same job.

## Correctness is necessary, not sufficient

```
go test ./...
```

This is green right now, before you touch anything, because a copy of correct
code is correct. The suite pins `Optimized`'s output to `Baseline`'s byte for
byte, including a 1000-entry production-shape input, and it must stay green
through every change you make. A faster function with different output is a
bug, not an optimisation. The exam is the gate below, not these tests.

## The workflow

Benchmark and profile commands run from `svc/` so the output files land next
to the code.

1. Record the baseline before any change:

   ```
   cd svc
   go test -run '^$' -bench . -benchmem -count=10 > before.txt
   ```

   `-count=10`, not 5: benchstat needs at least 6 samples to compute a
   confidence interval at the 0.95 level and prints `± ∞` on every row below
   that. Ten leaves margin for the machine to misbehave during one of them.

   On Windows PowerShell, `>` writes UTF-16, which benchstat cannot read; use
   `| Out-File -Encoding utf8 before.txt` instead, or run these commands in
   Git Bash or cmd.

2. Profile the function you are about to fix:

   ```
   go test -run '^$' -bench BenchmarkOptimized -benchmem -cpuprofile cpu.out
   go tool pprof -top cpu.out
   go tool pprof -http=:8081 cpu.out
   ```

   `-top` ranks functions by CPU time; the browser view has the flamegraph
   (View > Flame Graph), and the widest bar is the most expensive path. Expect
   runtime functions, `concatstrings` and `memmove`, not a name you wrote:
   the cost of `+=` is paid inside the runtime, so follow the callers up one
   frame to find your loop.

3. Fix `svc/optimized.go`. The profile already told you what to replace; the
   project page names the idiomatic tool. `go test ./...` must stay green.

4. Re-measure with identical flags:

   ```
   go test -run '^$' -bench . -benchmem -count=10 > after.txt
   ```

5. Prove it is not noise:

   ```
   benchstat before.txt after.txt
   ```

   benchstat is the one external tool in this project, and it stays optional
   for the machine check below. Note that the usual install line does not work
   on this lab's toolchain: `golang.org/x/perf` now requires Go 1.25 or newer,
   so `@latest` fails to build on Go 1.22. Pin a commit from before that bump:

   ```
   go install golang.org/x/perf/cmd/benchstat@400946f43c82
   ```

   A p-value above 0.05 means the difference could be variance; more `-count`
   runs or a bigger win. Read the `BenchmarkBaseline` row as well as the
   `BenchmarkOptimized` one: you did not edit `baseline.go`, so it is a control
   group, and it should report `~`. If it reports a significant change instead,
   the two files were measured under different conditions and the drift
   contaminates every row in the table.

6. The gate, the machine check, from the lab root:

   ```
   cd ..
   go test -tags gate -run TestGate ./...
   ```

   `TestGateSpeedup` first pins `Optimized`'s output to `Baseline`'s byte for
   byte at n=0, 1, and 1000 entries, because `-run TestGate` skips the
   correctness suite and a fast function with the wrong output is a bug, not a
   pass. Then it reruns `BenchmarkBaseline` and `BenchmarkOptimized` in one
   process and fails unless your ns/op is at least 40% below the baseline and
   your allocs/op at least 50% below, printing both measured deltas either way.
   The thresholds are relative on purpose: both sides move with your machine,
   the ratio does not. Run it without `-race`; the detector would measure its
   own instrumentation.

## Profile it live

Step 01 of the project profiles the running service the way you would in
production, over HTTP from a separate admin port. Three terminals:

```
go run .
go run ./loadgen -n 10000 -c 50
go tool pprof -http=:8081 "http://127.0.0.1:6060/debug/pprof/profile?seconds=30"
```

Start the load first, then take the profile while it runs: pprof samples for
30 seconds, and an idle service produces an empty profile with
`Total samples = 0`. The service renders through `Optimized`, so the
flamegraph shows your code, slow before the fix and flat after it.

The `-http` view needs a browser and Graphviz. The same data prints to a
terminal with `-top` (ranked functions) and `-list 'Optimized$'` (per source
line), which is what the project's verify steps use.

## Done means

- `go test ./...` green: it started green and stayed green.
- `go test -tags gate -run TestGate ./...` green.
- For the project's step 09 deliverable: the pprof output naming the source
  line, the `benchstat before.txt after.txt` comparison at `-count=10`
  including the `BenchmarkBaseline` control row, and one written paragraph
  explaining why the fix reduced copying and allocation.

The reference fix is `svc/solution.go`, sealed behind the `solution` build
tag so `go test` always grades your file. Do not open it until the gate is
green.
